Данный документ детализирует функциональные и нефункциональные требования к проекту бюрократического агента.


## 1. Цели и Область (Scope)

| **Тип** | **Описание** |
| --- | --- |
| Основная цель | Создать веб-платформу, позволяющую принимать заявки от граждан с агентом-бюрократом на бэкэнде. |
| Включено (In Scope) | Аутентификация (простая), отправка заявок (форма + документы), просмотр состояния заявок, получение обработной связи по заявкам, обработка заявок на бэкэнде (метаданные и документы).  |
| Исключено (Out of Scope) | Интерактивные сессии с AI-агентом, включение бюрократа в сессию, комплексные multi-stage сценарии, интеграции с реальными сервисами, … |

## 2. Требования к пользователю и аутентификации

### 2.1 Функциональные требования к аутентификации (FR-AUTH)

| ID | Требование | Роль | Примечания |
| --- | --- | --- | --- |
| FR-AUTH-100 | Система должна позволять пользователю входить в систему по email и паролю. | Citizen | JWT-токены для сессий. |
| FR-AUTH-101 | Система должна позволять пользователю выходить из аккаунта (logout). | Citizen | Инвалидация токена на клиенте. |
| FR-AUTH-102 | Система должна ограничивать доступ к заявкам только их владельцу. | Citizen | Пользователь видит только свои заявки. |

### **2.2 Функциональные требования к управлению пользователем (FR-USER)**

| **ID** | **Требование** | **Роль** | **Примечания** |
| --- | --- | --- | --- |
| FR-USER-200 | Пользователь должен иметь возможность просматривать свой профиль (email, дата авторизации). | Citizen | Базовая информация о пользователе. |

## **3. Функциональные требования к заявкам**

### **3.1 Создание и отправка заявок (FR-APP)**

| **ID** | **Требование** | **Роль** | **Примечания** |
| --- | --- | --- | --- |
| FR-APP-100 | Система должна позволять пользователю создавать новую заявку с выбором типа процедуры. | Citizen | Выбор из предопределённого списка процедур (например, "Получение паспорта", "Регистрация ИП"). |
| FR-APP-101 | Система должна позволять пользователю заполнять форму заявки с необходимыми полями. | Citizen | Поля зависят от типа процедуры (динамическая форма). |
| FR-APP-102 | Система должна позволять пользователю загружать документ к заявке (PDF, DOCX, JPG, PNG). | Citizen | Пока только один документ. Потом сделаю возможность множественного добавления. |
| FR-APP-103 | Система должна валидировать заполненную форму перед отправкой. | System | Проверка обязательных полей, формата данных (email, телефон, ИНН и т.д.). |
| FR-APP-104 | Система должна позволять пользователю отправлять заявку на обработку. | Citizen | После отправки заявка получает статус "На рассмотрении". |
| FR-APP-105 | Система должна генерировать уникальный номер заявки при создании. | System | Формат: APP-YYYYMMDD-XXXXX. |

### 3.2 Просмотр и управление заявками (FR-APP)

| **ID** | **Требование** | **Роль** | **Примечания** |
| --- | --- | --- | --- |
| FR-APP-200 | Система должна позволять пользователю просматривать список своих заявок. | Citizen | Пагинация, сортировка по дате создания, фильтр по статусу. |
| FR-APP-201 | Система должна позволять пользователю открывать детальную страницу заявки. | Citizen | Просмотр всех данных заявки, загруженных документов, истории статусов. |
| FR-APP-202 | Система должна позволять пользователю отменять заявку до начала обработки. | Citizen | Заявка переходит в статус "Отменена". |
| FR-APP-203 | Система должна отображать текущий статус заявки. | Citizen | Статусы: "Черновик", "На рассмотрении", "В обработке", "Требуется дополнение", "Одобрена", "Отклонена", "Отменена". |

### **3.3 Обработка заявок AI-агентом (FR-AGENT)**

| **ID** | **Требование** | **Роль** | **Примечания** |
| --- | --- | --- | --- |
| FR-AGENT-300 | Система должна автоматически проверять заявку на полноту при отправке. | AI Agent | Проверка наличия всех обязательных полей и документов согласно чеклисту процедуры. |
| FR-AGENT-301 | Система должна адаптироваться под разные сценарии проверки на основе существующей документации бюрократических процессов | AI Agent | То есть мы не делаем статический workflow. Мы делаем агента, который прочитывает информацию относительно того или иного бюрократического процесса и делает те или иные действия на основе этого. |
| FR-AGENT-302 | Система должна автоматически извлекать данные из загруженных документов (OCR/парсинг). | AI Agent | Извлечение ФИО, дат, номеров документов из PDF/изображений. |
| FR-AGENT-303 | Система должна генерировать отчет о проверке заявки и отправлять обратную связь для пользователя | AI Agent | Список найденных проблем или подтверждение корректности. Также понятное описание того, что нужно исправить или дополнить. |
| FR-AGENT-304 | Система должна автоматически изменять статус заявки на основе результатов проверки | AI Agent |  |

## 5. Нефункциональные требования (NRF)

| ID | Описание | Категория | Примечания |
| --- | --- | --- | --- |
| NFR-001 | Система должно обрабатывать заявки фоново, чтобы человек не ждал ответа. | Performance | Асинхронная обработка, очереди задач (Celery/RQ). |
| NFR-002 | Все операции с персональными данными должны выполняться по HTTPS. | Security | SSL/TLS сертификаты. |
| NFR-003 | Загруженные документы должны храниться в зашифрованном виде. | Security |  |
| NFR-004 | Система должна логировать все действия пользователей и AI-агента. | Observability | Логи в файл/сервис (Sentry, CloudWatch). |
| NFR-005 | Периодически агент должен автоматически проходить автоматическую оценку. | Evaluation | Evaluation CI/CD. |
| NFR-006 | Интерфейс должен быть адаптивным (responsive) для мобильных устройств. | Usability | Mobile-first подход в дизайне. |
| NFR-007 | У нас невозможен спам запросами. | Security | Rate limiter. |
